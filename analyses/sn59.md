# Subnet 59 — Babelbit

> Analyzed: 2026-05-13 · live snapshot from chain (Finney) at analysis time

## TL;DR

Babelbit is a real, well-funded **low-latency speech-translation** subnet
run by a UK team out of the Yuma / Score ecosystem. The current mining
task is **predictive utterance completion** — given a partial sentence,
predict how it ends — scored on lexical similarity, semantic similarity
and *earliness*. Registration is **cheap** right now (~**0.0009 τ**
recycle on the snapshot below), but **all 256 UID slots are full** — new
registrations **evict the lowest-incentive UID** (no free slots). The
protocol still splits **80% Arena / 20% qualifying** in intent, but
**dashboard `top1_share` can read ~100%** when the on-chain `incentive`
vector has only one strictly-positive entry (see *Reading `top1_share`*
below — it is **not** the same thing as “one miner earns every τ of
subnet emission”). Great learning subnet; serious income only if you
can crack the Arena / top-qualifier game.

## What it actually does

Babelbit (`babelbit.ai`, UK-based BabelBit Ltd, part of the Score
ecosystem) is building real-time speech-to-speech translation. The
end-state is a "speak in language A, hear language B with sub-second
latency" pipeline; today the subnet only mines the **prediction** piece
that makes that latency possible.

Validators give each miner a partial multilingual utterance and the
miner returns the most likely completion. Miners are scored on three
axes:

| Axis | What it measures |
|---|---|
| Lexical similarity | string-level overlap with ground truth (BLEU-style) |
| Semantic similarity | embedding-distance — same meaning ≠ same words still scores |
| Earliness | how short the prefix was when you committed to a correct prediction |

Custom adequacy metrics (EATP, Lead, ACS) are documented in the repo;
an LLM-based judge for translation quality is on the near-term roadmap.

**Deployment model is hybrid:** Babelbit’s **Arena** track expects a
**Dockerized “chute”** on [Chutes](https://chutes.ai) and validator traffic
through that stack. The **qualifying / self-hosted** track in the official
[`babelbit_miner`](https://github.com/babelbit/babelbit_miner) repo is a
**FastAPI** server you expose yourself (`/v1/predict`) plus on-chain **axon**
IP/port registration. Check Discord / current round rules for which path
your UID should be on — both exist in the public miner README.

## Reward structure (confirmed from `babelbit_subnet` repo §5.2)

| Tier | Share of emission | Who gets it |
|---|---|---|
| **Qualifying** | 20% | every miner above threshold, proportional to score |
| **Arena**     | 80% | **winner-take-all per challenge**, drawn from top-7 qualifiers |

Arena eligibility rules:

- Must have won **at least one qualifying challenge in the last 7 days**.
- **7 arena slots** today (subject to change as the team evolves the mech).
- Slots are ranked by number of recent wins, then by score.
- The Arena prize is awarded **per challenge**, so over a week it
  rotates through the top contenders rather than locking onto one.

That's why **live `top1_share` can spike to ~100%** in the subnetscope /
Taostats UI — read the next section before interpreting it as “one miner
owns the whole subnet.” If you fall out of the top-7, you're competing for a
slice of the remaining 20% pool spread across the long tail.

## Reading `top1_share` (why it can show **~100%**)

Subnetscope (and similar tools) compute `top1_share` from the metagraph
**`incentive`** tensor: sort all **strictly positive** incentives, then
divide the largest by their sum.

On the **2026-05-13** Finney snapshot used for this refresh, **only UID 248**
(a **validator** with `validator_permit=True`) had **`incentive = 1.0`**;
**every miner UID showed `incentive = 0.0`** in the same read. That makes the
formula collapse to **100% top-1** even though it is **not** describing
“one miner captured all emissions.” It is mostly telling you **“only one
UID currently carries positive Yuma incentive weight in this tensor.”**

What to use instead when reasoning about **miner** inequality:

- **Emission / alpha paid over time** (Taostats charts, subnetscope *miner
  rewards* / emission views) — not the raw `incentive` concentration alone.
- **Your own UID’s emission** after you have been receiving weights for
  multiple tempos — new miners often sit at **zero** for a while.

Treat **`top1_share` as a volatile chain/UI signal**, not a stable product
spec.

## What miners run

- **Hardware:** CPU is enough for the *current* text-prediction task. A
  GPU mainly helps if you want to fine-tune your own predictor; pure
  inference of the reference model is light.
- **Stack:** Python; the `bb` CLI (installed via `uv`); the Chutes SDK
  for packaging your model as a Docker chute; a hotkey/coldkey pair for
  on-chain registration. Validators run a Postgres + Dockerized
  validator node — irrelevant unless you're validating.
- **Reference miner:** [`Babelbit-YY_02` on HuggingFace](https://huggingface.co/might2901/Babelbit-YY_02)
  is a working baseline. Read it to understand the I/O contract before
  you build your own.
- **Time to first reward:** hours after publishing your chute and
  registering, you should appear in the next validator round.

## Cost to operate a miner

✅ no paid API keys required · 💻 no GPU required · 📋 verified from repo

**Monthly recurring:** ~$25 – $120 · **One-time setup:** ~$15 – $30 (registration, varies with chain price)

### Monthly costs

| Item | Required | USD / month | Notes |
|---|:-:|---|---|
| Chutes container hosting | ✓ | $20 – $100 | miners are deployed as Chutes containers; pay per GPU-second of usage |
| VPS (operator coordinator) | ✓ | $5 – $20 | small Linux VPS to manage the deployment |

### One-time costs

| Item | Required | USD | Notes |
|---|:-:|---|---|
| Registration burn / recycle (one-time per UID) | ✓ | varies | **~0.0009 τ** on 2026-05-13 snapshot — always re-check live recycle |
| Chutes account setup | ✓ | $0 | free signup at chutes.ai |

> **Subscription required:** Chutes account (free signup, pay-per-use compute) when you use the Arena/chute path.
> On-chain rewards are **concentrated**; live `top1_share` can read **~100%** during certain metagraph snapshots — see *Reading `top1_share`* above.
> Real out-of-pocket is dominated by the Chutes container minutes when you use that path — keep your model lightweight (CPU-only inference is fine for the current text task) to stay near the low end of the range.

## Realistic earnings (refreshed)

Subnet emission is now **~52 τ/day** (TAO-equivalent on the snapshot formula
above). The **within-miner** split is not stable week-to-week; treat any
single-number “you will earn X” table as illustrative only.

| Outcome | Order of magnitude |
|---|---|
| Long tail / new UID before weights | often **~0** until validators assign flow |
| Qualifying tier, mid-pack | highly variable — check live Taostats |
| **Arena rotation, in top-7 on a winning day** | can dominate **most** of the miner-bound TAO during streaks |
| Arena consistent winner (rare) | can spike very high while the streak lasts |

Numbers are **not** guarantees. Read **emission charts** and your **UID’s
paid alpha**, not only `top1_share`.

## Validator scoring (what's live)

- Validators reach miners via **on-chain axon metadata** (IP:port for the
  **self-hosted FastAPI** path in `babelbit_miner`) **and/or** via **Chutes**
  routing for the **Arena / container** path — follow the current round
  rules in Discord; both deployment models appear in the official miner README.
- Validator outputs go to **Babelbit-operated services for score
  aggregation** — meaning final ranking is computed off-chain and only
  the resulting weights hit the chain. This is unusual; trust assumption
  is that the team aggregates honestly.
- `commit_reveal_weights_enabled = yes` — validators must commit weight
  hashes first, then reveal, which protects against last-block
  copying. Doesn't affect miners directly but it's a good sign for
  fairness.

## Honest assessment

**Pros:**

- Burn / recycle can be **extremely cheap** at low demand — re-check live
  values before budgeting.
- **256/256 UIDs** → registration is now **competitive eviction**, not a free slot.
- No GPU required to start on the **text** task.
- Long immunity window (~16 h) — bad submissions get a real chance to
  iterate before deregistration pressure.
- Strong, identifiable team (Matthew Karas / Josh Greifer / Tom Horner),
  Yuma Accelerator backing, real product roadmap.
- Active repo: `babelbit_subnet` is being committed to and the miner
  repo and reference HuggingFace model are public.

**Cons:**

- Reward shape remains **winner-take-all leaning** for the Arena slice;
  live **`top1_share` can print ~100%** during certain on-chain snapshots
  — interpret with the *Reading `top1_share`* section, not as a literal
  “one key earns every τ” statement.
- **Off-chain score aggregation** — you're trusting Babelbit's services
  to rank fairly. Lower than Bittensor's usual on-chain transparency.
- Chutes dependency adds an extra moving part (and an outage there
  takes you down) **when you use that deployment path**.
- Task is text-only today, but the project's *thesis* is speech — when
  the task swaps to streaming audio, your text-only model goes to zero.
  Plan to migrate.

## Getting started

1. Read the official subnet repo (validator + protocol docs):
   <https://github.com/babelbit/babelbit_subnet>
2. Read the miner repo for the actual mining bits:
   <https://github.com/babelbit/babelbit_miner>
3. Look at the reference HF model to understand the I/O:
   <https://huggingface.co/might2901/Babelbit-YY_02>
4. Join the Babelbit Discord (linked from `babelbit.ai`) and ask in
   `#mining` for the current base-miner config.
5. Wallet: hot+cold pair, fund the coldkey with a small **τ** buffer
   (recycle can be sub‑0.001 τ at low demand but **moves with the pool**).
6. Install `uv`, then `pip install` the `bb` CLI and the Chutes SDK.
   Build your model as a chute and publish it.
7. Run the base miner against the public validator endpoint **first**
   to confirm you score above the qualifying threshold *before*
   spending on registration.
8. `btcli subnet register --netuid 59 --wallet.name <cold> --wallet.hotkey <hot>`
   (when **256/256 full**, this **evicts** the lowest-incentive UID — check
   live docs / Discord for the exact eviction rule).
9. **Self-hosted:** `register_axon.py` + `bb server` on your public IP/port.
   **Chutes path:** publish the chute and wire metadata per Babelbit’s
   submission flow. Watch scores over the ~16 h immunity window.

## Roadmap (where the task goes next)

| Phase | Status | What it means for miners |
|---|---|---|
| Phrase / utterance completion (text) | **current** | what miners do today |
| Live text-to-text translation (streaming) | early 2026 | text-prediction skill carries over; add MT |
| Speech-in / text-out (streaming ASR) | mid-2026 | latency-critical ASR pipelines required |
| Full speech-to-speech | late 2026+ | end-to-end audio models or tuned cascade |

Mid-roadmap, the validator task will likely break the current text-only
miners. Treat sn59 as **active development**, not "set and forget".
